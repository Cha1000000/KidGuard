# Известные пределы защиты и обходы (веха 6.5)

> Честный разбор того, что KidGuard **не может** закрыть без Device Owner, и почему для нашего
> сценария (своя семья, ребёнок 11–14, ручная установка `.apk`, владелец не делает factory reset)
> это приемлемо. Составлено на шаге 6.5.

## Базовое ограничение: не Device Owner

KidGuard — обычное приложение + **Device Admin** + **Accessibility** + **SYSTEM_ALERT_WINDOW** +
**VpnService**, но **НЕ Device Owner** (его нельзя назначить без ADB/сброса при первичной настройке,
а мы ставим `.apk` вручную). Из этого следуют почти все пределы ниже.

## Что закрыто (веха 6)

- **Отключение VPN / Accessibility / деактивация Device Admin / удаление KidGuard** — перехватываются
  детским `AccessibilityService` (детект по заголовку окна) и накрываются PIN-оверлеем
  (`TYPE_ACCESSIBILITY_OVERLAY`, поверх защищённых экранов). Без родительского PIN не пройти (6.2, 6.3).
- **Удаление KidGuard** дополнительно требует деактивации Device Admin (система не даёт удалить, пока
  админ активен), а деактивация — под PIN.
- **Откат системного времени** (сброс дневного счётчика) — анти-отмотка (веха 2): дата не «откатывается»
  назад относительно ранее виденной.

## Что закрыть НЕЛЬЗЯ (и почему приемлемо)

1. **Safe mode (безопасный режим).** В нём отключаются ВСЕ сторонние приложения, включая KidGuard
   (accessibility/оверлей/VPN не работают). Запретить вход в safe mode без Device Owner нельзя.
   *Почему приемлемо:* в safe mode отключены и все игры/приложения ребёнка, аккаунты недоступны —
   пользоваться телефоном по назначению нельзя, стимул низкий. Выход из safe mode — перезагрузка,
   после которой контроль восстанавливается (автозапуск по `BOOT_COMPLETED`).

2. **Factory reset (сброс к заводским).** Удаляет KidGuard и все данные. Без Device Owner (и FRP)
   предотвратить нельзя. *Почему приемлемо:* владелец ([[Владимир]]) гарантирует не делать сброс сам;
   ребёнок 11–14 вряд ли пойдёт на сброс — теряются все его игры, аккаунты, фото. Это «ядерная» опция,
   не бытовой обход.

3. **ADB / режим разработчика + ПК.** Если ребёнок включит «Параметры разработчика» → USB-отладку и
   подключит телефон к компьютеру, он сможет удалить/отключить KidGuard через `adb`. *Почему
   приемлемо:* требует навыков, компьютера и физического кабеля; для целевой аудитории (обычный
   подросток своей семьи) — маловероятно. Защиту самих «Параметров разработчика» PIN-ом можно добавить
   позже тем же механизмом (детект экрана по заголовку), если понадобится — кандидат на доработку.

4. **Программный always-on VPN.** Форсить always-on + lockdown без Device Owner нельзя — родитель
   включает вручную при первичной настройке (мастер разрешений напоминает). После включения ребёнок
   не сможет отключить always-on без PIN (экран VPN-настроек под PIN, 6.2).

## Вывод

Для заявленного сценария (своя семья, доверенный владелец, ребёнок-неэксперт) критичных дыр,
делающих контроль бессмысленным, нет: бытовые обходы (отключить VPN/accessibility, удалить приложение,
отмотать время) закрыты PIN-ом и анти-отмоткой; оставшиеся пределы (safe mode, factory reset, ADB)
либо самоограничивают ребёнка (теряет свои данные/игры), либо требуют экспертных действий и ПК.

Полноценная защита от подготовленного обхода требует **Device Owner** — это отдельный путь (provisioning
через ADB/QR при настройке устройства), возможный в будущем, но за рамками текущего sideload-сценария.
См. [[KidGuard — концепция]].
