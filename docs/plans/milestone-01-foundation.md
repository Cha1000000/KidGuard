# Веха 1 — Каркас и фундамент (детальный план)

Обзор всех вех — `roadmap.md`. Концепция проекта — `[[KidGuard — концепция]]` во «втором мозге».

## Цель вехи

Заложить рабочий каркас приложения, на котором строятся все остальные вехи:
1. Единоразовый выбор роли (родитель / ребёнок) с фиксацией **навсегда**.
2. Навигационный каркас под обе роли (экраны-заглушки).
3. Мастер пошаговой выдачи разрешений на детском устройстве.
4. `AccessibilityService`, определяющий активное (foreground) приложение.

**Что НЕ входит в веху 1** (чтобы не расползлось): учёт времени, лимиты, реальная блокировка,
Google Sign-In, бэкенд, VPN. Здесь — только фундамент и проверка, что ключевые механизмы Android
нам доступны.

## Предусловия

- Каркас Empty Activity (Compose) уже создан, собирается (`./gradlew assembleDebug`).
- Разрешения объявляются в `AndroidManifest.xml` по мере добавления шагов.

## Архитектурное решение (согласовано 04.07.2026)

**Организация кода:** пока проект одномодульный (`app`), **пакетная** структура по Clean
Architecture внутри `app`, без преждевременной модуляризации:

```
ru.homelab.kidguard
├── core          // общие модели, утилиты, DI-модули
│   ├── data
│   ├── domain
│   └── ui        // тема, общие Compose-компоненты
├── feature
│   ├── onboarding   // выбор роли + мастер разрешений
│   ├── parent       // экраны родителя (заглушки)
│   └── child        // экраны ребёнка (заглушки) + сервисы контроля
└── platform      // AccessibilityService, DeviceAdmin, Foreground service
```

Разбиение на Gradle-модули (`core`/`parent`/`child`/`sync`) отложим до вехи 2, когда появится
реальный код исполнителя и границы устоятся.

**Логирование:** Timber (согласовано).

## Декомпозиция на шаги

Каждый шаг — самостоятельный, проверяемый, коммитится отдельно после отмашки.

### Шаг 1.1 — Базовая инфраструктура
- Подключить **Hilt** (DI): `@HiltAndroidApp`-класс Application, базовый модуль.
- Подключить **Navigation Compose**.
- Подключить **DataStore (Preferences)** — для хранения выбранной роли и флагов настройки.
- Завести пакетную структуру (см. выше). Настроить тему (оставить сгенерированную Material 3).
- **Критерий готовности:** приложение собирается и запускается, Hilt-граф поднимается,
  DataStore читается/пишется (проверка тестовым значением).

### Шаг 1.2 — Выбор роли и роутинг при старте
- Модель роли: `enum Role { PARENT, CHILD }` + хранение в DataStore (`selectedRole`, `setupDone`).
- Экран онбординга: выбор «Чей это телефон?» → сохранить роль. Пометка в UI, что выбор
  **единоразовый** (предупредить пользователя).
- Стартовый роутинг: при запуске читаем роль. Нет роли → онбординг; роль есть → граф этой роли.
  Повторного логина/логаута нет.
- **Критерий готовности:** первый запуск ведёт в онбординг; после выбора роли и перезапуска
  приложение открывает нужный граф без повторного вопроса.

### Шаг 1.3 — Навигационный каркас ролей (заглушки)
- Граф родителя: нижняя навигация/меню с экранами-плейсхолдерами (Дети, Правила, Статистика —
  пустые). Граф ребёнка: главный экран-плейсхолдер (Сегодня / Осталось времени — пустой).
- **Критерий готовности:** оба графа открываются, навигация между заглушками работает.

### Шаг 1.4 — Мастер выдачи разрешений (детская роль)
- Пошаговый экран (визард), проводящий через выдачу и **проверку статуса** каждого разрешения:
  1. **Usage Access** (`PACKAGE_USAGE_STATS`) — доступ к статистике использования.
  2. **Accessibility Service** — включение нашего сервиса.
  3. **Overlay** (`SYSTEM_ALERT_WINDOW`) — рисование поверх.
  4. **Device Admin** — активация (защита от удаления).
  5. **Отключение оптимизации батареи** (важно для HiOS/MIUI).
  6. **VPN consent** + инструкция включить Always-on VPN (текст-подсказка; сам VPN — веха 5).
- Каждый пункт: индикатор «выдано/не выдано» + кнопка «Выдать» → системный экран/диалог.
  После возврата — перепроверка статуса.
- **Критерий готовности:** на реальном/эмулированном устройстве мастер корректно определяет
  статус каждого разрешения и ведёт в нужный системный экран; отмечает выданные зелёным.

### Шаг 1.5 — AccessibilityService: детект активного приложения
- Скелет `KidGuardAccessibilityService` (`accessibilityServiceInfo`: типы событий
  `TYPE_WINDOW_STATE_CHANGED`). На событии определяем package активного приложения.
- Пока просто логируем (Timber/Logcat) текущий foreground-пакет.
- **Критерий готовности:** при переключении между приложениями в логах виден корректный
  package name активного приложения (включая системные и Play Market).

### Шаг 1.6 — Скелет Foreground Service
- `KidGuardForegroundService` с постоянным уведомлением (канал), автозапуск по `BOOT_COMPLETED`.
- Пока без логики учёта — только живёт в фоне (фундамент под веху 2) и переживает сворачивание.
- **Критерий готовности:** сервис стартует, держит уведомление, переживает сворачивание
  приложения; на Tecno (HiOS) проверяем, что не убивается сразу (тест — на вехе 6 подробнее).

## Definition of Done (веха 1)

- Приложение собирается и запускается; выбор роли работает и фиксируется.
- Оба ролевых графа открываются (с заглушками).
- Мастер разрешений корректно отражает статусы и ведёт в системные экраны.
- AccessibilityService определяет активное приложение (видно в логах).
- Foreground service живёт в фоне.
- Ничего из этого не блокирует пользователя реально — только фундамент (блокировка — веха 2).

## Риски и заметки

- **HiOS/MIUI** могут убивать фоновый сервис — полноценно проверяем на вехе 6, но скелет
  закладываем сейчас (foreground + boot receiver).
- **Порядок выдачи разрешений** важен для UX — Accessibility и Overlay требуют ручных шагов в
  системных настройках; мастер должен быть терпеливым и перепроверять статус.
- **Тестовое устройство:** часть механизмов (Accessibility детект, реакция на разрешения) стоит
  гонять на реальном Tecno или эмуляторе с Google APIs.

## Решения по вехе 1 (согласовано 04.07.2026)

- **Пакетная структура** (core / feature / platform) внутри `app` — принята.
- **Логирование** — Timber.
- **PIN** — целиком отложен на веху 6 (в вехе 1 ещё нечего защищать; заводить каркас заранее —
  спекулятивно). На вехе 6 сделаем и хранение, и применение.
