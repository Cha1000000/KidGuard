# Веха 3 — Пер-app лимиты

> Пересмотрено и согласовано с Владимиром 2026-07-07. Бонусное время **вынесено** в отдельную
> мини-веху (веха 3Б, см. ниже) — здесь его не делаем.

Обзор всех вех — `roadmap.md`. Зависит от **вехи 2** (учёт времени, блокировка, белый список).

## Цель вехи

**Лимиты на отдельные приложения:** родитель задаёт каждому приложению свой дневной лимит
(минут/день). Приложение блокируется при исчерпании **своего** лимита независимо от общего
дневного лимита. Настройка локальная (Room); удалённая выдача — веха 4.

**Не входит** в эту веху (сознательно вынесено):
- Бонусное время → **веха 3Б** (см. в конце).
- Детский UI с остатком по приложениям → отдельная веха (сейчас ребёнок видит только
  оверлей-блокировку + предупреждение из вехи 2).

## Что уже есть из вехи 2 (на что опираемся)

- `UsageRepository` — суммарный учёт: `screenTimeSeconds(date): Flow<Int>` + `addScreenTime(date, seconds)`.
  Room-таблица `screen_time` (дата → секунды).
- `PolicyRepository` — `dailyLimits`, `whitelist`, `setDailyLimit`, `setWhitelisted`.
- `LimitState` (`NoLimit`/`Remaining`/`Expired`) + `ObserveLimitStateUseCase` (общий дневной лимит).
- `shouldBlock(activePackage, limitState, whitelist, alwaysAllowed)` — чистое правило блокировки.
- `ScreenTimeTracker` — тик 15 c, пишет время только когда «пользователь активен»; **уже знает
  активный пакет** через `ForegroundAppMonitor.currentPackage`.
- `BlockingController` — combine(активный пакет, лимит, белый список) → оверлей + уход на домашний.
- Room `KidGuardDatabase` **version = 1**, `exportSchema = false`.

## Матрица приоритетов правил (зафиксировано однозначно)

Для активного приложения решение «блокировать?» вычисляется по порядку:

1. **`alwaysAllowed`** (само KidGuard + текущий лаунчер) → **никогда не блокируем**.
2. Иначе, если у приложения задан **личный (пер-app) лимит и он исчерпан** → **блокируем**
   (личный лимит — более точечное правило, бьёт даже белый список).
3. Иначе, если приложение в **белом списке** → **доступно** (личного лимита нет/не исчерпан,
   общий дневной игнорируем — так и было в вехе 2).
4. Иначе, если **общий дневной лимит исчерпан** → **блокируем**.
5. Иначе → **доступно**.

Короче: `alwaysAllowed` > пер-app лимит > белый список > общий лимит.

## Декомпозиция на шаги

### Шаг 3.1 — Модель пер-app лимитов (данные)
- Room: новая таблица `app_limits` (`package` PK → `minutes`). Room `version 1 → 2` + **миграция**
  (`CREATE TABLE`; писать `Migration(1,2)`, не destructive — учимся делать по-взрослому).
- `PolicyRepository`: добавить `appLimits: Flow<Map<String, Int>>` (пакет → минут) и
  `setAppLimit(packageName: String, minutes: Int?)` (`null` убирает лимит).
- Реализация в `PolicyRepositoryImpl` + новый DAO-метод(ы) в `PolicyDao` (или отдельный DAO).
- **Критерий:** пер-app лимиты сохраняются и читаются; миграция 1→2 проходит без потери данных.

### Шаг 3.2 — Поминутный учёт по приложениям
- Room: новая таблица `app_screen_time` (составной PK `date` + `package` → `seconds`).
- `UsageRepository`: добавить `appScreenTimeSeconds(date, packageName): Flow<Int>` и
  `addAppScreenTime(date, packageName, seconds)`.
- `ScreenTimeTracker`: в том же тике, где пишет суммарное время, **дополнительно** прибавлять
  секунды активному пакету (`ForegroundAppMonitor.currentPackage.value`). Суммарный учёт из вехи 2
  не ломать — он остаётся для общего лимита.
- **Критерий:** время по каждому приложению считается отдельно и корректно; суммарное — по-прежнему верно.

### Шаг 3.3 — Блокировка по пер-app лимиту
- Доменное решение расширить под матрицу приоритетов выше. Аккуратный вариант: новый чистый
  use case (напр. `ObserveBlockingDecisionUseCase`) или расширить `shouldBlock` так, чтобы для
  **активного пакета** учитывались и его личный лимит/расход, и общий.
- Личное состояние лимита приложения удобно посчитать по аналогии с `ObserveLimitStateUseCase`
  (личный расход `app_screen_time` за сегодня против `app_limits[package]`).
- `BlockingController`: т.к. личный расход зависит от активного пакета динамически — вероятно
  `flatMapLatest` по `currentPackage` (для активного пакета наблюдать его usage+limit), затем
  комбинировать с общим лимитом и белым списком. Реализацию оставляем Fable, направление — здесь.
- **Критерий:** приложение с исчерпанным личным лимитом блокируется, даже если общий не исчерпан;
  другие приложения доступны; белый список и `alwaysAllowed` ведут себя по матрице.

### Шаг 3.4 — Родительский UI (⚠️ вне зоны кодинг-агента)
- Экран настройки пер-app лимитов: список **запускаемых** приложений (LAUNCHER-activity, с иконкой
  и названием — как в Family Link), у каждого — задать/убрать личный лимит.
- Источник списка — `PackageManager` (query LAUNCHER intent); загрузчик приложений — новый кусок
  в `platform` (иконки/названия).
- **UI делаем по правилу проекта: сперва макеты → согласование с Владимиром → реализация.**
  Кодинг-агент (Fable) этот шаг **не трогает** — останавливается после 3.3 и возвращает управление.
- **Критерий:** родитель локально задаёт/убирает пер-app лимиты по списку приложений.

## Definition of Done (веха 3)

- Отдельные приложения блокируются по своим лимитам независимо от общего дневного.
- Матрица приоритетов (alwaysAllowed > пер-app > белый список > общий) работает предсказуемо.
- Пер-app учёт корректен, суммарный учёт вехи 2 не сломан, миграция Room 1→2 проходит.
- Родительский UI пер-app лимитов реализован (после согласования макетов).

## Проверка (как тестируем)

- **Юнит-тесты** доменного решения блокировки (матрица приоритетов — все ветки).
- **Эмулятор** (техника из вехи 2): вставить `app_limits` и `app_screen_time` через
  `run-as ... sqlite3`, проверить, что приложение с исчерпанным личным лимитом блокируется, а
  соседнее — нет; проверить пер-app учёт (тик прибавляет активному пакету).

## Веха 3Б — Бонусное время (отдельная мини-веха, ПОЗЖЕ)

Вынесено из вехи 3. Реализуем после того, как пер-app лимиты готовы и обкатаны.

- **Поведение (согласовано):** бонус **разовый на сегодня** + возможность **досрочной отмены**
  родителем. Без накопления/переноса на другой день.
- Модель гранта: тип (весь телефон / конкретное приложение), объём (минуты), действует до конца дня.
- Применение: активный грант временно снимает соответствующую блокировку; истекает в конце дня
  или при досрочной отмене.
- Приоритет: бонус встраивается в матрицу как временное послабление (снимает блок телефона/приложения).
- UI: выдача/отмена бонуса — по правилу макеты → согласование → реализация.
