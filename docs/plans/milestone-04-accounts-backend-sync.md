# Веха 4 — Аккаунты, бэкенд и синхронизация

> Пересмотрено и согласовано с Владимиром 2026-07-08. Первая веха с серверной частью.
> Зависит от вех 2–3 (есть локальная политика и статистика, которые синхронизируем).

Обзор всех вех — `roadmap.md`.

## Цель вехи

1. **Google Sign-In** (Credential Manager) на обеих ролях — профиль привязан к верифицированному Google-аккаунту.
2. Родитель **добавляет ребёнка по email**; родители равноправны.
3. **Бэкенд** на AdminVPS: единая политика ребёнка (single source of truth), синхронизация правил и статистики.
4. **Мгновенные обновления** (бонус, смена правил) — pull + WebSocket.

## Согласованные решения

- **Стек бэкенда:** Node.js + **Express 5 + better-sqlite3 + JWT + `ws`** (переиспользуем связку
  из FinKeeper24 — она отлажена на этом же AdminVPS). Порт **3003** (3002 занят финкипером).
  Модульная структура (`routes/` `services/` `middleware/`).
- **Push-стратегия:** база — **periodic pull** (детское устройство тянет политику при старте и
  раз в N минут — foreground-сервис уже работает 24/7); **WebSocket** — для мгновенности пока
  устройство онлайн. **FCM отложен** — вернёмся к нему, только если pull+WS окажется мало.
- **Разработка:** локально в `~/projects/KidGuard-server` (Node/Express), тесты на `localhost:3003`
  с временной БД. **Боевой сервер не трогаем**, пока не готово и не протестировано.
- **Репозиторий бэкенда:** отдельный **приватный** GitHub-репо (клиент KidGuard — публичный,
  серверную логику публично не выставляем).
- **Адрес на старте:** `http://157.22.172.217:3003` (по IP). Поддомен + nginx + HTTPS — позже.
- **Деплой:** по образцу `sync-to-deploy.sh` (rsync кода без секретов/БД → контролируемый push).
  Отдельная папка на сервере (напр. `/var/www/kidguard-server/`), отдельный Node-процесс.
  **Только по явной команде Владимира.** С FinKeeper не пересекаемся никак.

## Архитектура (клиент ↔ сервер)

```
Родитель (Android)  ─┐                          ┌─ БД (better-sqlite3):
  Google Sign-In      │   REST (Retrofit/OkHttp) │    users, parent_child,
  правит политику ────┼──────────────────────────┤    policies (JSON+updated_at),
  смотрит статистику  │   WebSocket (ws)          │    usage_stats
                      │                           │
Ребёнок (Android)   ──┘   pull политики + WS      └─ Node/Express (:3003)
  Google Sign-In          применяет в Room            верификация Google ID-token,
  шлёт статистику         (вехи 2–3 исполняют           выдача JWT, авторизация
  исполняет правила        офлайн)                       «свой ребёнок»
```

- **Единая политика ребёнка** — один документ на сервере; оба родителя редактируют его, ребёнок
  только читает и применяет локально. Локальный Room на детском устройстве — исполнитель (вехи 2–3),
  сервер — источник истины.
- **LWW (Last-Write-Wins)** по серверному `updated_at`: при конфликте одновременных правок побеждает
  последняя по времени сервера. Для «под себя» (2 родителя) этого достаточно.

## Схема БД (better-sqlite3)

- `users` — `id`, `google_sub` (уникальный ID Google, стабильный), `email`, `display_name`, `created_at`.
- `parent_child` — `parent_user_id`, `child_user_id` (M:N: два родителя ↔ один ребёнок; у родителя
  может быть несколько детей). PK (`parent_user_id`, `child_user_id`).
- `policies` — `child_user_id` PK, `data` (JSON: дневные лимиты по дням, пер-app лимиты, белый список,
  бонусы), `updated_at`, `updated_by` (кто из родителей правил). Версионируется для LWW.
- `usage_stats` — `child_user_id`, `date`, `package_name`, `seconds`. PK (`child_user_id`,`date`,`package_name`).
  Плюс суммарное экранное время за день.

> Политику храним **JSON-документом** (а не нормализованными таблицами): она целиком читается/пишется
> при синхронизации, LWW проще на уровне документа, структура повторяет доменные модели клиента.

## REST API (контракт)

Аутентификация: `Authorization: Bearer <JWT>` (кроме `/auth/google`). Авторизация: доступ к политике/
статистике ребёнка — только связанным родителям или самому ребёнку.

- `POST /auth/google` — тело `{ idToken }`. Сервер **верифицирует Google ID-token**, находит/создаёт
  `user` по `google_sub`, возвращает свой JWT + профиль.
- `GET /me` — текущий профиль.
- `POST /children` — тело `{ childEmail }`. Родитель связывает себя с ребёнком по email (ребёнок
  должен быть уже зарегистрирован). Возвращает связь.
- `GET /children` — список детей родителя.
- `GET /policy/:childId` — единая политика ребёнка (+ `updated_at`).
- `PUT /policy/:childId` — заменить политику; сервер ставит `updated_at`=серверное время (LWW),
  рассылает WS-сигнал ребёнку и второму родителю.
- `POST /usage/:childId` — ребёнок присылает статистику за день (батч).
- `GET /usage/:childId` — родитель смотрит статистику.
- `WS /ws` — авторизованное соединение; сервер шлёт `{type:"policy-changed", childId}` при правке.

## Декомпозиция на шаги

Порядок: сперва серверный фундамент (по твоему выбору), потом клиент.

### Шаг 4.0 — Проектирование и каркас сервера
- Завести `~/projects/KidGuard-server` (приватный репо), Express 5 + better-sqlite3 + JWT + ws,
  структура `routes/services/middleware`, `.env` (секреты — вне git), скрипт локального запуска.
- Создать схему БД (миграции/инициализация), заглушки эндпоинтов, health-check.
- **Критерий:** сервер поднимается локально на `:3003`, БД инициализируется, `/health` отвечает.

### Шаг 4.1 — Аутентификация (сервер + клиент)
- Сервер: `POST /auth/google` — верификация Google ID-token (библиотека google-auth-library),
  создание user, выдача JWT; JWT-middleware.
- Клиент: **Credential Manager (Sign in with Google)** на обеих ролях, получение ID-token, обмен на
  JWT, хранение JWT в DataStore. Сетевой слой (**Retrofit + OkHttp** — подключаем впервые).
- **Требуется от Владимира:** Google Cloud project — OAuth client ID (Web — для верификации на
  сервере; Android — для клиента). Настройка в консоли Google Cloud.
- **Критерий:** обе роли логинятся через Google; клиент получает JWT; сервер верифицирует token.

### Шаг 4.2 — Связывание родитель ↔ ребёнок
- Сервер: `POST /children` (по email), `GET /children`; таблица `parent_child`; проверки доступа.
- Клиент (родитель): UI добавления ребёнка по email, список детей.
- **Критерий:** двое родителей привязаны к одному ребёнку, оба его видят.

### Шаг 4.3 — Синхронизация единой политики
- Сервер: `GET/PUT /policy/:childId` (LWW по `updated_at`).
- Клиент: маппинг локальной политики (Room вех 2–3) ↔ серверный JSON; **pull при старте + периодически**
  (в foreground-сервисе) и применение в Room; родитель — `PUT` при правке.
- **Критерий:** правка родителя А доходит до ребёнка и родителя Б; на детском телефоне один
  согласованный набор правил.

### Шаг 4.4 — Синхронизация статистики
- Сервер: `POST/GET /usage/:childId`.
- Клиент: ребёнок батчами шлёт статистику; родитель видит её в UI (экран «Статистика» — заглушка из вехи 1).
- **Критерий:** родитель видит актуальную статистику ребёнка.

### Шаг 4.5 — WebSocket-push
- Сервер: `WS /ws`, рассылка `policy-changed` при `PUT /policy`.
- Клиент: WebSocket-клиент (OkHttp) в foreground-сервисе; по сигналу — немедленный pull политики.
- **Критерий:** выданный родителем бонус применяется на детском телефоне почти сразу (пока онлайн).

### Шаг 4.6 — Деплой на боевой AdminVPS (по команде Владимира)
- Деплой-клон + скрипт (по образцу `sync-to-deploy.sh`), папка `/var/www/kidguard-server/`,
  процесс-менеджер (pm2/systemd), проверка на боевом. Поддомен/nginx/HTTPS — отдельно позже.
- **Критерий:** сервер работает на боевом, два реальных устройства синхронизируются.

## Definition of Done

- Обе роли входят через Google; профиль привязан к аккаунту.
- Родитель добавляет ребёнка по email; равноправные родители управляют единой политикой.
- Правила и статистика синхронизируются между двумя телефонами через AdminVPS.
- WebSocket доставляет срочные изменения (бонус/правила) почти мгновенно, когда устройство онлайн.

## Тестирование

- **Сервер:** локально (`localhost:3003`), временная БД; юнит/интеграционные тесты эндпоинтов
  (auth, доступ «чужой ребёнок» запрещён, LWW).
- **Клиент:** два эмулятора (родитель + ребёнок) против локального сервера; проверка полного цикла
  (вход → связывание → правка политики → долетело до ребёнка → статистика обратно → бонус по WS).

## Безопасность (важно для серверной вехи)

- Секреты (`GOOGLE_CLIENT_ID`, `JWT_SECRET`, ключи) — только в `.env`, **вне git**.
- Каждый запрос: валидный JWT + проверка, что пользователь имеет доступ именно к этому ребёнку.
- HTTPS — при заведении поддомена (на старте по IP — только для локальной обкатки/доверенной сети).

## Открытые вопросы / на согласование по ходу

- Точный формат JSON-документа политики (согласуем при шаге 4.3, отталкиваясь от доменных моделей).
- Частота periodic pull (компромисс батарея/свежесть) — подберём на шаге 4.3.
- Формат батча статистики и как часто ребёнок её шлёт — шаг 4.4.
- Регистрация ребёнка до связывания: ребёнок должен один раз войти на своём устройстве, чтобы
  появиться в БД, — согласуем UX на шаге 4.2.
