## Цель вехи

1. **Google Sign-In** (Credential Manager) на обеих ролях — профиль привязан к верифицированному Google-аккаунту.
2. Родитель **добавляет ребёнка по email**; родители равноправны.

## Согласованные решения

## Аутентификация ролей (пересмотрено 2026-07-08)

**Ключевое решение:** родитель и ребёнок аутентифицируются **по-разному**.

- **Родитель — Google Sign-In** (Credential Manager): управляет политикой, добавляет детей,
  приглашает второго родителя.
- **Ребёнок — pairing-код, БЕЗ Google-входа.** Родитель в приложении создаёт ребёнка (задаёт имя)
  и получает короткий **6-значный код**; на детском устройстве ребёнок вводит этот код и устройство
  привязывается, получая свой device-токен.

**Почему не Google-вход у ребёнка:** реальные детские аккаунты обычно под **Family Link (supervised)**,
а Google намеренно блокирует OAuth-вход таких аккаунтов в сторонние несертифицированные приложения
(наше — sideload, не из Play Store). Pairing-код обходит это полностью и удобнее ребёнку (не нужен
пароль Google). Проверено на практике при разработке 4.1.

## Архитектура (клиент ↔ сервер)

```
Родитель (Android)  ─┐                          ┌─ БД (better-sqlite3):
  Google Sign-In      │   REST (Retrofit/OkHttp) │    users (родители),
  создаёт ребёнка+код │                          │    children, parent_child,
  правит политику ────┼──────────────────────────┤    pairing_codes,
  смотрит статистику  │   WebSocket (ws)          │    policies (JSON+updated_at),
                      │                           │    usage_stats
Ребёнок (Android)   ──┘   pull политики + WS      │
  вводит pairing-код      применяет в Room        └─ Node/Express (:3003)
  шлёт статистику         (вехи 2–3 исполняют         verify Google ID-token (родитель),
  исполняет правила        офлайн)                     pairing (ребёнок), выдача токенов,
                                                       авторизация «свой ребёнок»
```

- **Единая политика ребёнка** — один документ на сервере; оба родителя редактируют его, детское
  устройство только читает и применяет локально. Room на детском устройстве — исполнитель (вехи 2–3),
  сервер — источник истины.
- **LWW (Last-Write-Wins)** по серверному `updated_at`: при конфликте одновременных правок побеждает
  последняя по времени сервера. Для «под себя» (2 родителя) этого достаточно.

## Схема БД (better-sqlite3)

- `users` — **родители**: `id`, `google_sub` (уникальный ID Google), `email`, `display_name`, `created_at`.
- `children` — **дети** (без Google-аккаунта): `id`, `display_name` (задаёт родитель), `avatar`
  (индекс аватарки из набора в `:app`), `created_at`, `paired_at` (когда устройство привязалось;
  NULL — код ещё не введён).
- `parent_child` — `parent_user_id`, `child_id` (M:N: два родителя ↔ один ребёнок; у родителя может
  быть несколько детей). PK (`parent_user_id`, `child_id`).
- `pairing_codes` — `code` (6 цифр) PK, `child_id`, `created_at`, `used_at` (NULL — активен).
  **Бессрочный до ввода** (не истекает по времени); одноразовый (гасится при привязке). У ребёнка
  один активный код — перевыпуск (`/pair-code`) заменяет старый.
- `co_parent_invites` — приглашение второго родителя: `child_id`, `invited_email`, `expires_at`,
  `accepted_at`. Пока приглашённый email ещё не входил — связь ждёт его первого Google-входа.
- `policies` — `child_id` PK, `data` (JSON: дневные лимиты, пер-app лимиты, белый список, бонусы),
  `updated_at`, `updated_by`. Версионируется для LWW.
- `usage_stats` — `child_id`, `date`, `package_name`, `seconds`. PK (`child_id`,`date`,`package_name`).

> Политику храним **JSON-документом**: целиком читается/пишется при синхронизации, LWW проще на
> уровне документа, структура повторяет доменные модели клиента (вехи 2–3Б).

## Токены

- **Родитель** — JWT с `{ userId }` (Google-вход, 30 дней; реализовано в 4.1).
- **Детское устройство** — device-JWT с `{ childId, kind: "child" }`, выдаётся при успешной привязке
  кода. Долгоживущий (устройство не должно «разлогиниваться»); отзыв — при отвязке (веха 6).

## REST API (контракт)

Аутентификация: `Authorization: Bearer <token>`. Авторизация: к политике/статистике ребёнка —
только связанные родители или само это детское устройство.

**Родитель (Google-токен):**
- `POST /auth/google` — `{ idToken }` → verify Google ID-token, find-or-create родителя, вернуть JWT + профиль. *(реализовано 4.1)*
- `GET /me` — текущий профиль родителя. *(реализовано 4.1)*
- `POST /children` — `{ name }`. Создать ребёнка, привязать к текущему родителю, сгенерировать
  pairing-код. Вернуть `{ child, code, expiresAt }`.
- `GET /children` — список детей родителя (+ статус `paired`).
- `POST /children/:childId/pair-code` — перевыпустить код (если истёк / повторная привязка).
- `POST /children/:childId/co-parent` — `{ email }`. Пригласить второго равноправного родителя.
  Если такой родитель уже зарегистрирован — сразу `parent_child`; иначе — `co_parent_invites`
  (привяжется при его первом входе).

**Детское устройство (pairing / device-токен):**
- `POST /device/pair` — `{ code }` (без auth). Проверить код → пометить `used_at`, `paired_at` →
  выдать device-JWT + `{ childId, name }`.

**Общее (родитель или детское устройство):**
- `GET /policy/:childId` — единая политика (+ `updated_at`).
- `PUT /policy/:childId` — заменить (только родитель); `updated_at` = серверное время (LWW); WS-сигнал.
- `POST /usage/:childId` — детское устройство шлёт статистику (батч).
- `GET /usage/:childId` — родитель смотрит статистику.
- `WS /ws` — авторизованное соединение; сервер шлёт `{ type: "policy-changed", childId }` при правке.

- **Критерий:** родитель создаёт ребёнка и получает код; детское устройство по коду привязывается;
  оба равноправных родителя видят одного ребёнка.

### Шаг 4.3 — Синхронизация единой политики
- **Сервер policy-agnostic:** хранит политику как непрозрачный JSON (`policies.data`) — не парсит
  и не валидирует структуру, только отдаёт/заменяет + ставит `updated_at` (серверное время, LWW) и
  `updated_by`. Формат документа — забота клиента.
- Сервер: `GET /policy/:childId` (доступ: связанный родитель ИЛИ само детское устройство) →
  `{ data, updatedAt }` (или `null`, если политики ещё нет); `PUT /policy/:childId` (только
  родитель) — заменить `data`, `updated_at = now` (LWW: последний PUT выигрывает).
- **Формат JSON-документа политики** (правила вех 2–3; бонусы — в 4.6):
  `{ "dailyLimits": {"MONDAY":90,…}, "appLimits": {"pkg":60,…}, "whitelist": ["pkg",…] }`.
- Клиент: маппинг локальной политики (Room вех 2–3) ↔ серверный JSON; **pull при старте +
  периодически** (в foreground-сервисе) и применение в Room; родитель — `PUT` при правке. Локальный
  `updatedAt` сравнивается с серверным, чтобы не перетирать более свежее.
- **Критерий:** правка родителя А доходит до ребёнка и родителя Б; на детском телефоне один
  согласованный набор правил.

### Шаг 4.4 — Синхронизация статистики
- Сервер: `POST/GET /usage/:childId`.
- Клиент: детское устройство батчами шлёт статистику; родитель видит её в UI (экран «Статистика» —
  заглушка из вехи 1).
- **Критерий:** родитель видит актуальную статистику ребёнка.

### Шаг 4.5 — Селектор активного ребёнка (мультидетность в UI родителя)
Согласовано 2026-07-10 (вариант «глобальный селектор»): один общий выбор ребёнка на всё
приложение родителя — вкладки «Правила» и «Статистика» показывают выбранного.

- **`activeChildId` в DataStore** (`kidguard_sync`): выбранный ребёнок переживает перезапуск;
  если не задан или ребёнка больше нет — берётся первый из списка `/children`.
- **Room-политика родителя = кэш активного ребёнка** (решение против `child_id` в схеме —
  оверинжиниринг для 2–3 детей): при переключении ребёнка — pull его политики с сервера и
  `replaceAll` в Room (механизм шага 4.3). Переключение требует сети; без сети — сообщение об
  ошибке, остаёмся на текущем ребёнке.
- **SyncRepository:** активный ребёнок для push/pull берётся из `activeChildId` вместо костыля
  «первый из списка»; при смене — немедленный pull перед разрешением push (защита от перетирания
  политики нового ребёнка старым Room-содержимым).
- **UI (по правилу проекта — сперва макет, согласование, потом код):** чип с аватаркой и именем
  в шапке «Правил» (на «Статистике» чип уже есть — станет кликабельным) + bottom sheet
  «Чей профиль показать?» со списком детей. При одном ребёнке — чип без стрелки, sheet не
  открывается.
- **Критерий:** у родителя два ребёнка; переключение чипом меняет и правила, и статистику;
  правка лимита уходит именно выбранному ребёнку; выбор переживает перезапуск приложения.

### Шаг 4.6 — WebSocket-push + бонусы («Дополнительное время»)
Согласовано 2026-07-11: **бонусы — часть policy-документа** (вариант «в составе политики»),
отдельного endpoint нет. Смягчение LWW-гонки двух родителей — WS-pull у обеих ролей.
- Сервер: `WS /ws?token=<jwt>` (device- или parent-JWT; невалидный — закрытие соединения);
  при `PUT /policy` рассылка `{type:"policy-changed", childId}` устройству ребёнка и всем его
  родителям; ping/pong для чистки мёртвых соединений. Сервер остаётся policy-agnostic —
  секцию `bonuses` не парсит.
- Формат документа: `+ "bonuses": [{"date":"ГГГГ-ММ-ДД","packageName":""|"pkg","minutes":N}]`
  (записи за сегодня; `packageName:""` — бонус телефона, как в Room-таблице `bonus_grants`).
- Клиент: бонусы включаются в push/pull и канонизированный снапшот; выдача бонуса родителем =
  обычная правка политики. WebSocket-клиент (OkHttp) в детском foreground-сервисе И в
  родительской петле: по `policy-changed` — немедленный pull; reconnect с бэкоффом;
  периодический pull остаётся страховкой.
- **Критерий:** выданный родителем бонус применяется на детском телефоне почти сразу (пока онлайн);
  правка политики вторым родителем прилетает первому без перезахода.

- **Критерий:** сервер работает на боевом, два реальных устройства синхронизируются.

## Definition of Done

- Родитель входит через Google; детское устройство привязывается **pairing-кодом** (без Google-входа).
- Родитель добавляет ребёнка (имя → код); равноправные родители (второй — по email) управляют единой политикой.
- Правила и статистика синхронизируются между телефонами родителя и ребёнка через AdminVPS.
- WebSocket доставляет срочные изменения (бонус/правила) почти мгновенно, когда устройство онлайн.

## Тестирование

- **Сервер:** локально (`localhost:3003`), временная БД; юнит/интеграционные тесты эндпоинтов
  (auth, pairing: валидный/истёкший/использованный код, доступ «чужой ребёнок» запрещён, LWW).
- **Клиент:** два эмулятора (родитель + ребёнок) против локального сервера; полный цикл
  (родитель вошёл → создал ребёнка+код → ребёнок ввёл код → правка политики → долетело до ребёнка →
  статистика обратно → бонус по WS). **Детскому эмулятору Google-аккаунт не нужен.**

## Безопасность (важно для серверной вехи)

- Секреты (`GOOGLE_CLIENT_ID`, `JWT_SECRET`, ключи) — только в `.env`, **вне git**.
- Каждый запрос: валидный токен (родительский JWT или device-JWT) + проверка доступа именно к этому ребёнку.
- Pairing-код: одноразовый (гасится при вводе), генерируется криптостойко; не логировать в открытом
  виде. Не истекает по времени, но перевыпуск инвалидирует прежний — родитель может отозвать
  «утёкший» код, создав новый.
- HTTPS — при заведении поддомена (на старте по IP — только для локальной обкатки/доверенной сети).
