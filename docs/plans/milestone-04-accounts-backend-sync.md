# Веха 4 — Аккаунты, бэкенд и синхронизация (черновик)

> ⚠️ **ЧЕРНОВИК И РАМОЧНЫЙ ПЛАН.** Самая «обсуждаемая» веха: стек бэкенда, протокол и push пока
> не выбраны. **В начале вехи — отдельная сессия проектирования серверной части.** Здесь —
> только контур и точки принятия решений.

Обзор всех вех — `roadmap.md`. Зависит от **вех 2–3** (есть политика и статистика для синхронизации).
Сервер — на **AdminVPS** (доступ у агента есть; напомнить детали в начале вехи).

## Цель вехи

1. Google Sign-In (Credential Manager) — профиль привязан к верифицированному Google-аккаунту.
2. Родитель добавляет ребёнка по email; родители равноправны.
3. Бэкенд: **единая политика ребёнка** (single source of truth), синхронизация правил и статистики.
4. Push-канал для мгновенных обновлений (изменение правил, бонус).

## Декомпозиция на шаги

### Шаг 4.1 — Google Sign-In на клиенте
- Credential Manager (Sign in with Google) на обеих ролях; получение ID-token.
- Детская роль: вход аккаунтом ребёнка → регистрация детского профиля. Ручной ввод email для
  своего профиля НЕ используется.
- **Критерий:** обе роли логинятся через Google, клиент получает верифицированный email + token.

### Шаг 4.2 — [ПРОЕКТИРОВАНИЕ] Бэкенд на AdminVPS
- **Выбрать стек сервера** (обсудить: Node/Express как в FinKeeper24, или иное) — открытый вопрос.
- Схема БД: родители, дети, связь родитель↔ребёнок по email (многие-ко-многим), **единая политика
  ребёнка**, статистика использования.
- REST API: аутентификация (верификация Google ID-token на сервере), CRUD политики, приём статистики.
- **Критерий:** сервер поднят, API отвечает, ID-token верифицируется.

### Шаг 4.3 — Связывание родитель ↔ ребёнок
- Родитель добавляет ребёнка, вводя его Google-email. Сервер связывает. Без pairing-кодов.
- Оба равноправных родителя могут привязаться к одному ребёнку и видеть/редактировать его.
- **Критерий:** двое родителей привязаны к одному ребёнку, оба видят его.

### Шаг 4.4 — Синхронизация единой политики
- Детское устройство тянет единую политику (pull + по push-сигналу) и **применяет локально**
  (правила исполняются офлайн — из вех 2–3).
- Любой родитель редактирует одну и ту же политику; изменения доходят до второго родителя и до ребёнка.
- Конфликты одновременного редактирования — **Last-Write-Wins** по метке времени (как в HabitTracker).
- **Критерий:** изменение правила родителем А доходит до ребёнка и родителя Б; на детском телефоне
  один согласованный набор правил.

### Шаг 4.5 — Синхронизация статистики
- Детское устройство отправляет статистику использования на сервер; родитель видит её в UI.
- **Критерий:** родитель видит актуальную статистику ребёнка.

### Шаг 4.6 — Push-канал
- Мгновенная доставка обновлений (бонус, смена правил). **Выбрать канал** (обсудить: FCM?) —
  открытый вопрос.
- **Критерий:** выданный родителем бонус применяется на детском телефоне почти сразу.

## Definition of Done

- Обе роли входят через Google; профиль привязан к аккаунту устройства.
- Родитель добавляет ребёнка по email; равноправные родители управляют единой политикой.
- Правила и статистика синхронизируются между двумя телефонами через AdminVPS.
- Push доставляет срочные изменения.
