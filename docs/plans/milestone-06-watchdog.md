# Watchdog — самодиагностика контроля (веха 6)

> Статус: **план утверждён Володей 2026-07-16**, реализация начата.
> Закрывает открытый вопрос плана вехи 6: «Watchdog/самодиагностика при "зависании" контроля —
> решить, нужен ли». Ответ: нужен, повод — реальный случай ниже.

## Зачем (мотивация)

16.07 на детском эмуляторе контроль **не работал 26 минут, и никто об этом не узнал**:
accessibility-сервис был выключен (слетел после очистки данных приложения), `ForegroundAppMonitor`
не получал активное приложение, движок учёта не тикал. При этом:

- **детский экран бодро показывал «2 ч осталось из 2 ч»** — как будто всё в порядке;
- у родителя в Статистике просто перестали расти цифры — молча, без признака поломки;
- заметили случайно, глазами.

Это и есть «зависание контроля»: система в целом жива, но **защиты фактически нет**, а обе стороны
считают, что всё хорошо. На реальном HiOS (веха 6В) риск выше: вендорный менеджер автозапуска умеет
убивать foreground-сервисы целиком.

## Что ловим — две РАЗНЫЕ задачи

| | Что произошло | Кто может заметить |
|---|---|---|
| **(А) Разрешение отвалилось, сервис жив** | Выключили accessibility / отозвали usage access, overlay, Device Admin, VPN | **Сам сервис** — `PermissionsManager.isGranted()` уже всё это умеет |
| **(Б) Сервис убит целиком** | HiOS/MIUI выгрузил процесс; приложению «очистили данные»; телефон выключен | **Никто изнутри** — докладывать некому. Ловится только по молчанию |

Ключевое следствие: (А) решается докладом, (Б) — только отсутствием доклада. Поэтому **один и тот же
механизм закрывает оба**: ребёнок регулярно шлёт «я жив + вот моё здоровье»; молчание = (Б),
доклад с флагами = (А).

## Решения (Володя, 2026-07-16)

- **Ловим оба: (А) + (Б).**
- **Уведомляем ТОЛЬКО родителя.** Ребёнку не показываем: сообщение «контроль не активен» —
  это прямая подсказка «защиты нет, можно гулять». Детский экран не трогаем вообще.
- **Порог молчания: факт всегда, красное — после 12 ч.** «Последняя связь: N назад» показываем
  нейтрально при любом значении; красная плашка — только если молчание > 12 ч. Так ночной простой
  и разряженный телефон не превращают плашку в шум, который перестают замечать. Проблема (А) при
  этом ловится мгновенно и порога не требует вовсе.
- **Место в UI: плашка на карточке ребёнка** (вкладка «Дети», рядом со статусом «Привязан»), тап →
  лист с деталями и подсказкой, что делать. Баннеров на других экранах не делаем.
- **Когда всё хорошо — не показываем НИЧЕГО** (уточнение 16.07.2026). Карточка выглядит как
  сейчас («Привязан»), «последняя связь: N назад» на неё не выносим — при нормальной работе это
  лишняя строка. Факт последней связи виден в bottom-sheet по тапу.
- **«Что делать» — только текст, без кнопки.** Кнопка «открыть настройки» тут бесполезна:
  чинить нужно на телефоне РЕБЁНКА, а родитель смотрит со своего. Поэтому инструкция словами.
- **Частота heartbeat: 15 мин, на существующем `childSyncLoop`.** Новых таймеров не заводим —
  цикл и так ходит на сервер с этой периодичностью, расход батареи не меняется.

## Архитектура

### Принцип: сервер остаётся policy-agnostic

Веха 6 сознательно не трогала сервер (PIN спрятали в policy-документ). Здесь правки сервера
**неизбежны** — канал «ребёнок → родитель» существует только через него, — но держим их
минимальными и тупыми: сервер **хранит и отдаёт непрозрачный JSON**, не интерпретируя его.
Что именно значат флаги — знает только клиент.

### Канал

Ребёнок **уже** ходит на сервер каждые 15 минут (`CHILD_PULL_INTERVAL_MS` в `SyncRepositoryImpl`,
цикл `childSyncLoop()` поднят из `KidGuardForegroundService`). Вешаем heartbeat на этот цикл —
новый таймер не нужен.

```
ребёнок (каждые 15 мин + при старте сервиса)
   → POST /device/health  { health: {...} }   [device-JWT]
   → сервер: children.last_seen_at = now, children.health_json = <как прислали>
родитель (GET /children)
   → { id, name, avatar, paired, lastSeenAt, health }
   → UI решает, что показать
```

### Сервер (минимум)

1. **Миграция**: `ensureColumn('children', 'last_seen_at', 'last_seen_at TEXT')` и
   `ensureColumn('children', 'health_json', 'health_json TEXT')`. Хелпер уже есть в
   `db/connection.js` — на боевой БД отработает сам при рестарте, руками ничего не делать.
2. **`POST /device/health`** (`authenticateDevice`, только свой `childId`): пишет `last_seen_at =
   nowIso()` и `health_json` как есть. Тело не валидируем по смыслу — только что это объект.
3. **`GET /children`** (`listChildren`): добавить в ответ `lastSeenAt` и `health` (распарсенный JSON).

### Клиент — ребёнок

- Новый `DeviceHealthReporter` (в `:platform`): собирает `PermissionsManager.isGranted()` по всем
  `DevicePermission` + признак «accessibility реально забинден».
- Отправка из `childSyncLoop()`: при старте и на каждом тике (15 мин).
- **Ничего не показываем ребёнку.**

### Клиент — родитель

- `Child` (домен) получает `lastSeenAt: Instant?` и `health: DeviceHealth?`.
- `ChildrenScreen`: на карточке ребёнка — предупреждающая плашка, если
  (а) есть невыданные критичные разрешения, либо (б) `lastSeenAt` старее порога.
  Текст конкретный: «Не работает: Спец. возможности» / «Не выходил на связь 4 часа».
- Тап по плашке → лист с деталями и подсказкой, что делать.

## Порог «молчания» — решено

Выбран вариант «факт всегда, красное после 12 ч» (см. Решения). Обоснование: наивный порог даёт
ложные тревоги — телефон ребёнка ночью выключен или без интернета, и утром родитель увидел бы «не
выходил на связь 9 часов», хотя всё в порядке. Плашка, которая врёт каждое утро, перестаёт работать.

По итогам обкатки на реальном Tecno порог можно ужесточить — цифра в одном месте (`STALE_THRESHOLD`).

## Шаги (дробим — правило проекта)

- **W1 — сервер:** миграция + `POST /device/health` + `lastSeenAt`/`health` в `GET /children`.
  Тесты сервера (там уже 61 тест). Критерий: `curl` пишет health, `GET /children` его отдаёт.
- **W2 — ребёнок:** `DeviceHealthReporter` + отправка из `childSyncLoop`. Критерий: выключил
  accessibility на эмуляторе → через тик на сервере видно флаг.
- **W3 — родитель (домен/данные):** `Child.lastSeenAt` + `Child.health`, парсинг в `ChildRepositoryImpl`.
  Критерий: юнит-тест парсинга.
- **W4 — родитель (UI):** плашка на карточке + лист с деталями. **Сперва макет, потом код**
  (правило проекта). Критерий: воспроизвели сегодняшний случай — родитель видит «Не работает:
  Спец. возможности».
- **W5 — e2e:** на двух эмуляторах: (а) выключить accessibility → плашка у родителя;
  (б) убить процесс ребёнка → через порог плашка «не выходил на связь».

## Definition of Done

- Родитель видит, что контроль на устройстве ребёнка сломан, — и **что именно** сломано.
- Ребёнку ничего не сообщается.
- Ложных тревог при обычном ночном простое нет.
- Сервер не интерпретирует health — только хранит и отдаёт.

## Чего этот watchdog НЕ делает (и почему)

- **Не чинит сам.** Включить accessibility программно нельзя без Device Owner — это по определению
  ручное действие пользователя. Максимум — подсказать родителю, что нажать.
- **Не защищает от «очистить данные».** Отдельная дыра: очистка данных сбрасывает роль (приложение
  уходит в онбординг) и разрешения. Экран «Хранилище» в настройках сейчас **не** под PIN — ключевые
  слова детекта его не ловят. Кандидат в отдельную задачу вехи 6А; здесь только фиксируем факт.
