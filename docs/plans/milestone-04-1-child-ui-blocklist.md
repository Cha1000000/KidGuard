# Веха 4.1 — Детский UI + запрещённые приложения

> Статус: **утверждена Владимиром 2026-07-11** (шаги 4.1.1–4.1.3; шаг 4.1.4 «Попросить ещё
> времени» — отложен в отдельную будущую веху). Создана по требованию Владимира: детская
> половина приложения и «Полная блокировка нежелательных приложений» (ключевая фича концепции)
> выпали из вех 2–4 при детализации роадмапа. Реализуется **перед вехой 5 (VPN)**.

## Ресёрч аналогов (кратко)

- **Google Family Link (детская сторона):** ребёнок видит дневной лимит и остаток времени
  (через Digital Wellbeing), получает уведомление незадолго до блокировки; после лимита
  устройство блокируется, кроме «Unlimited time apps»; может запросить у родителя ещё время.
- **Kaspersky Safe Kids (детское приложение):** нарочито минималистично — сколько времени
  потрачено / какой лимит, и ответы родителя на **запросы доступа** («попросить разрешение»
  к заблокированному приложению/сайту — главная детская фича).
- **Общие best-practices** (Qustodio/Boomerang/обзоры): дружелюбный тон вместо «полиции
  экранного времени», прозрачность правил для ребёнка, крупный индикатор остатка времени,
  видимые категории «всегда доступные»/«заблокированные».

Выводы для нас: детский экран — минималистичный, «остаток времени» крупно + честный список
правил; киллер-фича аналогов «запрос времени/доступа» ложится на наш механизм бонусов
(родитель выдаёт «Дополнительное время»), но требует канала запросов на сервере.

## Шаг 4.1.1 — Список приложений ребёнка на сервере (фундамент) ✅ (12.07.2026)

> Выполнено: сервер — child_apps + PUT/GET /apps/:childId (задеплоено);
> клиент — публикация списка в childSyncLoop (снапшот против лишних отправок),
> экраны «Лимиты приложений»/«Всегда доступные» на серверном списке активного ребёнка,
> буквенный кружок-fallback, состояния «загрузка»/«список не получен».
> Доработка по решению Володи 12.07: детское устройство шлёт и ИКОНКИ (WebP 96×96
> base64) — сервер хранит их в child_apps.icon (идемпотентная миграция, json-лимит 5mb,
> 72/72 теста), родитель показывает серверную иконку (приоритет: серверная → локальная →
> буквенный кружок). Проверено e2e на эмуляторах с боевым сервером, включая приложение,
> которого нет у родителя (иконка отрисовалась с сервера).

**Проблема (вскрыта при планировании):** экраны «Лимиты приложений» и «Всегда доступные»
у родителя сейчас показывают приложения, установленные на РОДИТЕЛЬСКОМ телефоне
(`InstalledAppsProvider` читает локальный PackageManager). В многоустройственном мире (веха 4)
это неверно: правила задаются приложениям ребёнка, которых у родителя может не быть.

- Детское устройство отправляет на сервер список установленных **запускаемых** приложений
  (`packageName` + человекочитаемое имя) — при старте синк-петли и далее при изменениях
  (раз в цикл, если изменился канонический снапшот списка).
- Сервер: таблица `child_apps(child_id, package_name, label)`, `PUT /apps/:childId` (device),
  `GET /apps/:childId` (родитель). Policy-agnostic не нарушаем — это отдельный ресурс.
- Родительские экраны «Лимиты приложений», «Всегда доступные» (и новый «Запрещённые»)
  переключаются на серверный список приложений активного ребёнка (иконки — берём локальную,
  если приложение есть и у родителя, иначе — буквенный кружок как в «Статистике»).
- **Критерий:** родитель видит список приложений именно детского устройства.

## Шаг 4.1.2 — Запрещённые приложения (blocklist из концепции)

- Policy-документ: `+ "blockedApps": ["pkg", …]`.
- Родительский UI: карточка **«Запрещённые»** на вкладке «Правила» (иконка «щит/стоп»),
  внутри — список приложений ребёнка с чекбоксами (поиск, заблокированные вверху) —
  по образцу «Всегда доступные».
- Детский исполнитель: матрица приоритетов расширяется —
  `alwaysAllowed > **blocked** > пер-app лимит > белый список > общий дневной лимит`
  (запрет бьёт всё, кроме «всегда доступных»; KidGuard сам себя не блокирует).
- Оверлей блокировки — с формулировкой «Приложение недоступно» (текст согласуем на макете).
- **Критерий:** отмеченное родителем приложение блокируется на детском устройстве сразу
  (WS-доставка уже есть), включая Play Market и системные.

## Шаг 4.1.3 — Детский главный экран «Сегодня»

Замена заглушки. Содержимое (сверху вниз):
- Приветствие с именем и аватаркой ребёнка (имя/аватар приходят при привязке; добавить
  в ответ `/device/pair` и хранить в DataStore, обновлять при pull政ики? — уточнение на реализации).
- **Крупный индикатор остатка времени** на сегодня: кольцевой прогресс + «осталось 1 ч 23 мин
  из 2 ч» (или «Сегодня без лимита»); активный бонус отдельной строкой «+30 мин от родителей».
- Статус контроля: «Телефон под присмотром KidGuard» (+ работоспособность разрешений —
  бейдж «что-то отключено», если accessibility/оверлей потеряны — задел на веху 6).
- Раздел **«Мои правила»** (read-only прозрачность, как советуют аналоги): мини-списки
  «Всегда доступные», «С лимитом» (пер-app, с остатком), «Запрещённые».
- **Критерий:** ребёнок без вопросов к родителю понимает, сколько времени осталось и какие
  правила действуют; данные обновляются при pull/WS.

## ~~Шаг 4.1.4~~ «Попросить ещё времени» — ОТЛОЖЕН (решение Владимира 2026-07-11)

Запрос доп. времени от ребёнка (endpoint запросов + WS `time-requested` + UI) — в отдельную
будущую веху. В UI детского экрана задел не делаем.

## UI-порядок (правило проекта)

Макеты (HTML → PNG) на каждый новый экран: «Запрещённые» (родитель), детский «Сегодня»
(+ вид «время вышло», «без лимита»), изменённые «Лимиты приложений»/«Всегда доступные»
(серверный список) — согласование с Владимиром ДО кода.

## Распределение работ

Сервер (4.1.1 endpoints + тесты) — Sonnet-субагент по ТЗ; родительский/детский UI и
интеграция — Opus/Fable по макетам; матрица блокировки — Fable (критичная логика + тесты).

## Критерий вехи (DoD)

Родитель со своего телефона видит приложения ребёнка, запрещает любое (вкл. Play Market) —
на детском оно блокируется мгновенно; ребёнок на своём экране видит остаток времени, бонус
и свои правила. Всё работает против боевого AdminVPS.
