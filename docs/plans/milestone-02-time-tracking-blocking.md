# Веха 2 — Учёт времени и блокировка

Обзор всех вех — `roadmap.md`. Пересмотрен и уточнён перед стартом (06.07.2026).
Зависит от вехи 1 (Accessibility-детект, разрешения, DataStore, ForegroundAppMonitor).

## Цель вехи

Реализовать сердце родительского контроля на детском устройстве:
1. Точный учёт **реального экранного времени**.
2. Дневной лимит (по дням недели).
3. Белый список «всегда доступных» приложений.
4. Оверлей-блокировка по истечении лимита + предупреждение заранее.

Правила пока хранятся и настраиваются **локально** (синхронизация между устройствами — веха 4).

## Решения (согласовано с Володей 06.07.2026)

- **Foreground-сервис** из вехи 1 (шаг 1.6) не был реализован — веху закрыли на шаге 1.5.
  Создаём его теперь, в начале вехи 2 (там он и наполняется движком учёта).
- **Модуляризация — делаем сейчас**, пока кода мало (дешевле переносить, ниже риск потерять;
  мультимодульность — плюс для публичного портфолио). Структура — см. ниже.
- **Тикер учёта** — умеренный интервал (~10–15 с) при активном экране + реакция на screen on/off.
- **Граница «дня»** — локальная дата (`LocalDate`), сброс в полночь; перевод времени назад не
  даёт «отмотать» накопленное (базовая анти-отмотка; полноценная защита от обходов — веха 6).

## Шаг 2.0 — Модуляризация (отдельная ветка `refactor/modularization`)

Рефакторинг без изменения функционала — отдельной веткой и мёржем (чистая история). Слоевое
разбиение, устойчивое на все вехи (не feature-модули — их границы ещё не устоялись):

```
:app        UI и точка входа: MainActivity, KidGuardApp(@HiltAndroidApp), навигация,
            тема, feature-экраны (onboarding/parent/child). Зависит от core, data, platform.
:core       Чистый домен: модели (Role, DevicePermission, будущая Policy…), интерфейсы
            репозиториев, use cases. Минимум Android-зависимостей.
:data       Реализации: Room, DataStore, (позже network), реализации репозиториев. → :core
:platform   Системные интеграции Android: AccessibilityService, DeviceAdminReceiver,
            PermissionsManager, ForegroundAppMonitor, будущие ForegroundService/OverlayManager/
            VpnService. → :core
```

- Настройка: convention-плагин или общий блок для android/kotlin/hilt в каждом модуле;
  Hilt работает по модулям (модули с DI подключают hilt-android + ksp).
- Перенос существующего кода вехи 1 по модулям, сборка/тесты/эмулятор после переноса.
- **Критерий:** проект собирается в мультимодульной структуре, флоу вехи 1 работает как прежде.

## Шаг 2.1 — Foreground-сервис + автозапуск (перенос 1.6)

- `KidGuardForegroundService` (в `:platform`) с постоянным уведомлением (канал), тип сервиса.
- Автозапуск по `BOOT_COMPLETED` (BootReceiver). Пока без логики учёта — живёт в фоне.
- **Критерий:** сервис стартует, держит уведомление, переживает сворачивание и перезагрузку.

## Шаг 2.2 — Доменная модель и Room

- Room (в `:data`). Сущности: политика ребёнка (дневной лимит по дням недели), белый список
  (пакеты «всегда доступны»), накопленное экранное время за день (день → минуты).
- Доменные модели + интерфейсы репозиториев в `:core`, реализации в `:data`.
- **Критерий:** данные читаются/пишутся, схема мигрируется.

## Шаг 2.3 — Движок учёта реального экранного времени

- В foreground-сервисе — тикер (~10–15 с), инкремент **только когда**: экран включён
  (`PowerManager.isInteractive`) И разблокирован (`KeyguardManager` / `ACTION_USER_PRESENT`) И
  есть активное приложение (из `ForegroundAppMonitor`). Реакция на `ACTION_SCREEN_ON/OFF`.
- Музыка при погашенном экране — НЕ считать. Накопленное писать в Room (устойчиво к перезапуску).
- **Критерий:** экранное время растёт только при активном использовании; фоновая музыка и
  погашенный экран время не расходуют.

## Шаг 2.4 — Дневной лимит

- Сравнение накопленного за день с лимитом (учёт дня недели). Сброс по локальной дате в полночь.
- **Анти-отмотка:** перевод времени назад не уменьшает накопленное за текущий день.
- Состояние «время вышло» / «осталось N минут».
- **Критерий:** при достижении лимита система переходит в «время вышло»; перевод времени назад
  не сбрасывает счётчик.

## Шаг 2.5 — Оверлей-блокировка + белый список

- Когда активное приложение НЕ в белом списке И время вышло → блокирующий оверлей
  (`SYSTEM_ALERT_WINDOW`, `OverlayManager` в `:platform`) + возврат на home. Белый список доступен.
- **Критерий:** после лимита всё, кроме белого списка, перекрыто оверлеем; телефон/SMS/контакты работают.

## Шаг 2.6 — Родительский UI (локально)

- Экран настройки дневного лимита (по дням недели).
- Управление белым списком: список установленных приложений (`PackageManager`) с отметками.
- **Критерий:** родитель задаёт лимит и белый список — применяются на устройстве.

## Шаг 2.7 — Предупреждение перед истечением

- За N минут до лимита — оверлей/уведомление «осталось N минут».
- **Критерий:** предупреждение появляется заранее, ребёнок не блокируется внезапно.

## Definition of Done

- Проект в мультимодульной структуре; флоу вехи 1 работает.
- Foreground-сервис живёт в фоне и переживает перезагрузку.
- Экранное время считается корректно (реальное использование, без фоновых процессов).
- Дневной лимит по дням недели работает; по истечении — блокировка всего, кроме белого списка.
- Родитель локально настраивает лимит и белый список; работает предупреждение.

## Открытые вопросы (на реализацию)
