# Дизайн: запрет сайтов (DNS-фильтрация) на детском устройстве

- **Дата:** 2026-07-19
- **Статус:** согласовано (брейншторм), ждёт ревью спеки → план
- **Источник идеи:** перенос в KidGuard возможностей самописного модуля родительского контроля
  на роутере Володи (OpenWrt, `luci-app-parental`): «Чёрный список ресурсов» + «Блокировать
  поиск Google». См. `[[Роутер Cudy (OpenWrt)]]` во «втором мозге».

## Контекст (зачем)

На роутере родительский контроль (чёрный список сайтов, блок google-поиска) работает **только
дома**, в сети роутера. Как только ребёнок уходит на мобильный интернет или чужой Wi-Fi —
защита исчезает. KidGuard живёт на самом телефоне, и его `VpnService` работает **везде** —
логично перенести блокировку сайтов в приложение, чтобы она ездила с ребёнком.

Итог: родитель со своего телефона задаёт **чёрный список доменов** и тумблер **«Блокировать
поиск Google»**; на детском устройстве это исполняется **DNS-фильтром** поверх нашего VPN.
Блокировка держится, **пока родитель не снимет галку** (не завязана на дневной лимит).

## Решения (из брейншторма с Володей)

1. **Механизм — DNS-фильтрация** (по имени хоста), точнее IP-резолва роутера.
2. **Типы записей — только домены** (ввод домена или URL; URL нормализуем в хост, путь
   отбрасываем). Голые IP в v1 не поддерживаем (для них нужен пакетный дроп — позже, если надо).
3. **Обход через DoH/Private DNS** — в v1 **не боремся жёстко**: делаем DNS-прокси (ловит
   обычный DNS — большинство случаев), а про DoH/Private DNS честно пишем в подсказке (аналог
   оговорки «для CDN блок может быть неполным» на роутере). Порт 853/DoH-эндпоинты НЕ режем.
4. **Матчинг:**
   - чёрный список — по **суффиксу**: запись `vk.com` блокирует `vk.com`, `m.vk.com`, `api.vk.com`;
   - тумблер «Блокировать поиск Google» — **узко и точно**: только `google.com` и `www.google.com`
     (НЕ `mail.google.com`=Gmail, НЕ Play). Отдельный exact-match, не суффикс.
5. **Область действия** — всё детское устройство (как per-device на роутере), не пер-приложение.
6. **Размещение UI** — на экране **Правила** карточка **«Запрет сайтов»** под «Запрет приложений»,
   ведёт на новый экран. Вёрстка — по мотивам роутерного экрана, в нашей glassmorphism-системе.
7. **Хранение** — в policy-документе (сервер policy-agnostic, **не трогаем**).

## Архитектура и потоки данных

### 1. Модель и хранение (policy-документ, без сервера)

Политика уже возит `blockedApps`/`whitelist`/`appLimits`/PIN в непрозрачном JSON. Добавляем:
- `blockedSites: List<BlockedSite>`, где `BlockedSite(domain: String, enabled: Boolean)`
  (галка = заблокирован; снятая = временно разрешён, но не удалён).
- `blockGoogleSearch: Boolean`.

Затрагивается: `PolicyRepository` (+ flow `blockedSites`, `blockGoogleSearch`, + сеттеры),
`PolicyRepositoryImpl` (Room), policy-DTO в `data/.../network/PolicyApi.kt` + `replaceAll`/маппинг
синка. **Сервер и его схема не меняются** (документ непрозрачен).

**Нормализация домена** (при добавлении): в нижний регистр, срезать схему (`http(s)://`),
путь/порт/query, завершающую точку. Хранить хост как есть (`m.vk.com` остаётся `m.vk.com`).
Матч запроса `H` блокируется, если `H == entry` или `H` заканчивается на `.entry`.

### 2. Исполнение — DNS-прокси в VpnService (детское устройство)

Сейчас `KidGuardVpnService` — blackhole (tun не читается; блок пер-приложение через
`addDisallowedApplication`). Добавляем **режим DNS-фильтра**:

- В tun объявляем **виртуальный DNS-сервер** (`Builder.addDnsServer(DNS_VIRTUAL_IP)`) и
  заворачиваем в tun **только** его (`addRoute(DNS_VIRTUAL_IP, 32)`) — split-tunnel: DNS-запросы
  системного резолвера идут к нам, **весь остальной трафик — напрямую**.
- Читаем UDP:53-запросы из tun, парсим QNAME, сверяем с активными правилами (`blockedSites` с
  `enabled=true` по суффиксу + google-домены, если `blockGoogleSearch`):
  - совпало → отвечаем **NXDOMAIN** (или A `0.0.0.0`) → домен недоступен;
  - не совпало → форвардим на реальный upstream через **protected**-сокет
    (`VpnService.protect()`), ответ пишем обратно в tun.
- TCP:53 — редкий фолбэк; в v1 UDP-достаточно (TCP отметить как edge/после).

Компоненты (новые в `platform/.../vpn/`): парсер IP/UDP/DNS-пакетов (минимальный, только то, что
нужно), `DnsFilter` (правила матчинга), цикл чтения tun + форвардинг.

### 3. Сосуществование с блокировкой по лимиту (веха 5)

`VpnController` уже держит VPN всегда и переключает режим по `limitState`:
- **лимит исчерпан** → прежний **blackhole** (default route + disallowed=whitelist): и так всё
  заблокировано, DNS-фильтр не нужен;
- **время доступно** → **DNS-фильтр** (split-tunnel: только DNS в tun, остальное напрямую), если
  список/google-блок активны; иначе — как сейчас pass-through.

Режим и набор правил пересобираются реактивно (как сейчас `vpnDisallowedFor`) при смене
`limitState`/политики.

### 4. UI — новый экран «Запрет сайтов»

- **Правила** (`RulesScreen`): новая карточка «Запрет сайтов» под «Запрет приложений» →
  навигация на `BlockedSitesScreen`.
- **`BlockedSitesScreen`** (по мотивам роутера, в нашем стиле):
  - сверху — строка-тумблер **«Блокировать поиск Google»** с пояснением (ломает лазейку
    «поиск в HiOS Launcher → браузер»; Play/Gmail не трогает);
  - ниже — **«Чёрный список»**: поле ввода домена + кнопка «Добавить»; список записей, у каждой
    галка (вкл/выкл) и «−» (удалить); подсказка с честной оговоркой про DoH/Private DNS.
- **Сперва макет** в `docs/ui-concepts/site-blocking/` → согласование с Володей → потом код
  (правило проекта).

## Главный технический риск (проверить ранним спайком)

**Взаимодействие с always-on lockdown (веха 5.4).** DNS-фильтр в режиме «время доступно» —
split-tunnel: в tun только DNS, остальное идёт напрямую. При включённом системном **lockdown**
(«блокировать соединения без VPN») неотёсанный (не-DNS) трафик может **дропаться** → интернет
сломается в этом режиме. Без lockdown split-tunnel работает штатно.

**План действий:** первым шагом реализации — **on-device спайк**: поднять split-tunnel DNS-tun,
проверить (а) фильтрацию DNS, (б) что остальной трафик ходит, при lockdown ON и OFF. По итогу
выбрать: (1) v1 работает при lockdown OFF, документируем; (2) если критично при lockdown ON —
фолбэк к полному форвардингу трафика (тяжелее, tun2socks-подобно) — решаем по факту спайка.

## Известные ограничения (документируем, как «CDN» на роутере)

- **DoH/Private DNS (DoT)** обходят фильтр (Chrome и часть приложений; системный Private DNS).
- **TCP:53** в v1 не обрабатываем (edge).
- **Голые IP** в списке не поддерживаются (только домены/URL→хост).
- Взаимодействие с lockdown — см. риск выше.

## Проверка (end-to-end)

- Родитель задаёт домен (напр. `vk.com`) и включает «Блокировать поиск Google» → синк → на
  детском устройстве `vk.com`/`m.vk.com` и `google.com` не открываются в браузере, а `mail.google.com`
  (Gmail) и Play работают.
- Снятие галки у записи → домен снова доступен без удаления из списка.
- Проверка на детском эмуляторе + (позже) на реальном телефоне Олега.
- Спайк по lockdown (см. риск) — обязательный ранний шаг.

## Вне scope (v1)

- Голые IP, пакетный дроп по IP.
- Жёсткое противодействие DoH/DoT (порт 853, списки DoH-эндпоинтов).
- Пер-приложение область (только на всё устройство).
- TCP:53, DNSSEC-тонкости.

## Затрагиваемые слои (обзор, детали — в плане)

- `core`: модель `BlockedSite`, интерфейс `PolicyRepository` (+flow/сеттеры), правила матчинга
  (`DnsFilter`/матчер — по возможности в core как чистая логика).
- `data`: `PolicyRepositoryImpl` (Room), policy-DTO + синк (`replaceAll`, маппинг).
- `platform`: `KidGuardVpnService` (режим DNS-фильтра: парсер пакетов + прокси), `VpnController`
  (выбор режима), спайк по lockdown.
- `app`: карточка на `RulesScreen`, новый `BlockedSitesScreen` + ViewModel, строки, макет.
- **Сервер:** без изменений.
