# Дизайн: системные приложения в родительских пикерах (Запрещённые / Лимиты / Всегда доступные)

- **Дата:** 2026-07-18
- **Статус:** согласовано (брейншторм), готово к плану реализации
- **Источник:** обратная связь Володи по полевой обкатке (телефон Олега). Скриншот экрана
  «Статистика»: реально используемые системные приложения (`com.transsion.smartpanel`,
  `com.android.systemui`, `com.transsion.hilauncher`, `com.google.android.gms.supervision`,
  `com.android.settings` и т.п.) в Статистике видны, но в пикерах выбора приложений — нет.

## Контекст (зачем)

На родительских экранах **Запрещённые**, **Лимиты приложений** и **Всегда доступные** список
выбора строится только из **запускаемых** приложений детского устройства: детская часть
публикует на сервер лишь те приложения, у кого есть `CATEGORY_LAUNCHER`-активность
(`PlatformInstalledAppsSource.launchableApps()`, `platform/.../apps/PlatformInstalledAppsSource.kt:33`).
Системные приложения без launcher-иконки в этот список не попадают.

При этом экран **Статистика** показывает реально использованные пакеты (по-приложенчный
экранный тайм), включая системные. Володя хочет иметь возможность **ставить лимиты и запрещать
и такие приложения тоже** (например, `smartpanel`, всплывающие панели вендора).

Ожидаемый результат: во всех трёх пикерах доступны и системные приложения, которые ребёнок
реально использовал; критичные для работы телефона помечены предупреждением; системные вынесены
в отдельную категорию.

## Решения (из брейншторма с Володей)

1. **Источник списка:** к текущему списку добавляются системные пакеты, которые ребёнок
   **реально использовал** (есть экранное время). Ровно то, что видно в Статистике — без шума
   из сотен неиспользуемых системных пакетов.
2. **Безопасность:** критичные для работы устройства пакеты (`systemui`, дефолтный лаунчер,
   сам KidGuard) **не скрываем и не делаем неактивными** — показываем как обычные выбираемые
   строки, но с **значком-предупреждением** (жёлтый треугольник с восклицательным знаком).
3. **Группировка:** системные приложения выносим в отдельную категорию **«Системные»** и
   показываем **ниже** обычных приложений.
4. **Обратная совместимость (критично):** у Володи и Олега установлены текущие версии
   приложения; обновить детскую часть на телефоне Олега можно только через ~неделю. Изменения
   сервера — строго **аддитивные**, старые клиенты не должны сломаться.

## Архитектура и потоки данных

Фича сквозная — затрагивает детскую часть, транспорт (DTO + сервер) и родительский UI.

### 1. Детское устройство — источник объединённого списка + флаги

Собираем список как **`запускаемые ∪ реально использованные`** и вычисляем на каждом приложении
два флага (детское устройство универсально знает и свой лаунчер, и systemui — не подгоняем под
конкретную модель):

- `isSystem` — из `ApplicationInfo.FLAG_SYSTEM`. Управляет категорией «Системные».
- `isRisky` — пакет входит в «всегда разрешённые» (сам KidGuard + дефолтный лаунчер, та же логика,
  что в `BlockingController.alwaysAllowed`, `platform/.../overlay/BlockingController.kt:39`) **или**
  это `com.android.systemui`. Управляет значком-предупреждением.

Реализация:
- `InstalledAppsSource` (core) — расширить контракт: помимо `launchableApps()` уметь резолвить
  метаданные (label + иконка + `isSystem`) для произвольного набора пакетов. Вариант: новый метод
  `resolveApps(extraPackages: Set<String>): List<AppInfo>`, возвращающий запускаемые + резолв
  переданных пакетов (использованных), с проставленными флагами.
- `PlatformInstalledAppsSource` (platform) — реализация через `PackageManager`
  (`getApplicationInfo`/`getApplicationLabel`/`getApplicationIcon`, `FLAG_SYSTEM`). `isRisky`
  считаем здесь же по набору «всегда разрешённых» + systemui.
- `SyncRepositoryImpl.pushInstalledApps` (`data/.../sync/SyncRepositoryImpl.kt:225`) — передаёт
  в источник набор использованных пакетов (у него уже есть `usageRepository.appScreenTimeByPackage`,
  ср. `pushUsage`, `:248`). Снапшот-дедупликация (`LAST_SENT_APPS`) сохраняется, но теперь снапшот
  учитывает и флаги (изменение флага = переотправка).

### 2. Транспорт — DTO + доменные модели

- `ChildAppDto` (`data/.../network/AppsApi.kt:10`) — добавить `isSystem: Boolean = false`,
  `isRisky: Boolean = false` (значения по умолчанию → старый серверный ответ без полей парсится).
- `AppInfo` (core, `AppInfo.kt:8`) и `InstalledApp` (`app/.../rules/InstalledAppsProvider.kt:20`) —
  добавить те же два поля; `ChildAppsProvider.loadActiveChildApps()` (`:40`) их пробрасывает.

### 3. Сервер `/home/racer/projects/KidGuard-server` (аддитивно, деплой первым)

- **Миграция** (идемпотентно, по образцу колонки `icon`, `src/db/connection.js:124`,`:135`):
  ```js
  ensureColumn('child_apps', 'is_system', 'is_system INTEGER NOT NULL DEFAULT 0');
  ensureColumn('child_apps', 'is_risky',  'is_risky  INTEGER NOT NULL DEFAULT 0');
  ```
- `appsService.replaceApps` — INSERT c новыми колонками; при отсутствии флага в теле → `0`.
- `appsService.normalizeApps` — принять опциональные `isSystem`/`isRisky` (булевы, дефолт `false`;
  нестрогая валидация, как у `icon`).
- `appsService.getApps` — вернуть `isSystem`/`isRisky` (0/1 → boolean).
- `test/apps.test.js` — дописать кейсы: старое тело без флагов (дефолты), новое тело с флагами,
  round-trip GET.
- **Порядок деплоя:** сервер деплоим ПЕРВЫМ. Он совместим со старыми клиентами (старый детский PUT
  без флагов → колонки `0`; старый родительский GET → лишние поля игнорируются, `NetworkModule.kt:30`
  `ignoreUnknownKeys = true`). До обновления детской части у Олега фича «спит» (системные пакеты
  ещё не публикуются) — деградация мягкая, ничего не ломается.

### 4. Родительский UI — группировка + значок-предупреждение

Три экрана-пикера: **Запрещённые** (`BlockedAppsScreen`), **Лимиты приложений**
(`AppLimitsScreen`), **Всегда доступные** (`WhitelistScreen`). Общий паттерн отображения списка:

- Разбить список на две группы: обычные (`!isSystem`) — сверху; системные (`isSystem`) — ниже,
  под заголовком-разделителем **«Системные»**.
- В строке приложения с `isRisky` — значок ⚠️ (жёлтый треугольник с восклицательным знаком) рядом
  с названием. Строка остаётся полностью выбираемой (можно запретить/лимитировать/добавить в белый
  список).
- Заголовок «Системные» и (при необходимости) tooltip/подпись у значка — через `strings.xml`
  (без хардкода; правило проекта).
- **UI — сперва макет.** Layout сгруппированного списка + вид значка предупреждения → показать
  Володе макет и согласовать ДО кода (правило «сперва макеты»).

Если общая логика группировки/значка повторяется в трёх экранах — вынести в переиспользуемый
composable (напр. `AppPickerList`/`AppPickerRow` в `feature/parent/rules/`), не дублировать.

## Разбиение на этапы (реализация по шагам)

1. **Сервер:** миграция + сервис + тесты в `KidGuard-server`; прогнать тесты; **задеплоить**.
   Проверяемо независимо (тесты round-trip, curl PUT/GET).
2. **Детская часть + транспорт:** `InstalledAppsSource`/`PlatformInstalledAppsSource` (union +
   флаги), `ChildAppDto`/`AppInfo`/`InstalledApp`, `SyncRepositoryImpl.pushInstalledApps`.
   Проверка: детский эмулятор публикует системные использованные пакеты с флагами (лог/БД сервера).
3. **Макет родительского UI** (группировка «Системные» + значок) → согласование с Володей.
4. **Родительский UI:** сгруппированные пикеры + значок-предупреждение на всех трёх экранах.
   Проверка на родительском эмуляторе с боевым сервером.

## Проверка (end-to-end)

- Сервер: `npm test` в `KidGuard-server` зелёный (старое тело без флагов, новое с флагами,
  GET round-trip). Ручной curl PUT/GET подтверждает колонки.
- Обратная совместимость: **старая** сборка приложения (текущая, без правок) продолжает
  публиковать/читать `/apps` без ошибок против нового сервера.
- Детская часть: на эмуляторе ребёнка запустить пару системных приложений (накопить экранное
  время) → они появляются в опубликованном списке с `isSystem=1`, критичные с `isRisky=1`.
- Родитель: на всех трёх экранах системные — в категории «Системные» ниже обычных; на
  `systemui`/лаунчере/KidGuard — значок ⚠️, но строки выбираемы; можно задать запрет/лимит на
  системное приложение и оно применяется на ребёнке.

## Границы / вне scope

- НЕ переходим на «все установленные пакеты» — только использованные (решение из брейншторма).
- НЕ меняем enforcement-логику блокировки (она уже защищает лаунчер и сам KidGuard через
  `alwaysAllowed`); значок ⚠️ — предупреждение для родителя, а не запрет.
- Экран «Статистика» не трогаем (он и так показывает все использованные пакеты).
- Веха 6 остаётся закрытой; это доработка периода полевой эксплуатации.
